**Date:** 2026-04-09 13:28
**Scope:** `README.md`, `.gitignore`, `wordpress/` safe root files, `wordpress/wp-content/plugins/`, `wordpress/wp-content/themes/`, `wordpress/wp-content/mu-plugins/`

## Summary
Created a new standalone Git repository for `ruskyobchod.sk`, pulled the versionable WordPress code surface from Hostinger, and prepared it for publication as a public GitHub repo. The import intentionally excludes secrets, runtime artifacts, and obviously non-public premium bundles.

## Context & Problem
The user wants a durable handoff surface that another agent on a different machine can clone, inspect, change, and push back to the live Hostinger server. The live site had been edited directly on hosting by other parties, so there was no usable source-of-truth repository for the code currently running in production.

Because the target remote is a public GitHub repository, the import had to be selective. Publishing the entire web root would have leaked secrets and operational debris:
- `wp-config.php` contains credentials and salts
- uploads, caches, backups, and logs are runtime data, not source
- the live root contains one-off operational scripts and migration leftovers
- some plugin bundles are clearly premium/private and should not be republished in this public repo

## Decisions Made

### Scope the repo to the safe, versionable WordPress code surface
- **Chose:** import safe root bootstrap files, `.htaccess`, `wp-content/themes`, `wp-content/mu-plugins`, and `wp-content/plugins` with explicit exclusions.
- **Why:** this gives another agent the code that is most likely to be modified during maintenance while keeping the repo publishable and useful.
- **Alternatives considered:**
  - mirror the full web root — rejected because it would publish secrets, logs, backups, and one-off operational scripts
  - import only custom plugins/theme — rejected because live edits may also exist in non-custom plugins and that would miss too much of the actual deployable surface

### Exclude non-public and secret-bearing files
- **Chose:** exclude `wp-config.php`, generated plugin storage, runtime data directories, premium/private plugin bundles, and ad hoc operational scripts from version control.
- **Why:** the repo is being created as public. Safe publication matters more than perfect fidelity to every file on disk.
- **Alternatives considered:**
  - include everything and clean later — rejected because one accidental public push with secrets or premium code is enough
  - make the repo private instead — rejected because the user explicitly asked for a public repo

### Preserve the imported tree under a `wordpress/` directory
- **Chose:** put the imported site snapshot under `ruskyobchod/wordpress/`.
- **Why:** this keeps the repository root clean for repo metadata, worklogs, and future helper scripts while still mirroring the live site structure clearly.
- **Alternatives considered:**
  - make the web root the repo root — rejected because it mixes repo-control files with the imported site tree

## Architectural Notes
This repository is not an automated deployment setup yet. It is a baseline source snapshot intended to become the coordination surface for agent-driven work across machines. The likely operating model is:
- clone repo locally
- make small changes
- commit and push to GitHub
- deploy changed files to Hostinger over SSH with a deliberate sync step

The current tree is intentionally biased toward publishability and maintenance practicality over forensic completeness.
